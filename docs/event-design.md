# イベント設計

mendo におけるイベントの種類、BC 間のデータ取得方法、およびその使い分け。

## イベントの3つの種類

| 種類 | 中身 | 用途 | mendo での実装 |
|------|------|------|-------------|
| イベント通知 | 最小限（IDだけ） | 「起きた」を知らせる | — |
| 状態転送 | フルデータ or 差分 | データをコピーする | OrderBoard |
| 業務イベント | 業務の全情報 | 業務活動のモデル化 | domain/{集約}/events.go |

mendo の公開イベント（contract/）は「イベント通知 + 必要最小限のデータ」のハイブリッド。

## 内部イベント vs 公開イベント

| | 内部イベント | 公開イベント |
|---|---|---|
| 定義場所 | domain/{集約}/events.go | domain/contract/ |
| 中身 | 集約の内部構造が見える | 最小限の公開情報のみ |
| 用途 | 集約の状態変化を表現 | BC 間の連携 |
| 変更の影響 | 同じ BC 内のみ | 他 BC に影響する可能性 |

業務イベントをそのまま外部に公開すると密結合になる。contract/ で変換することで内部構造を隠蔽する。

## BC 間のデータ取得方法

### 方法B: イベントにデータを載せる（mendo で採用）

イベントに必要データを含めて、受信側が即座に処理する。

```
注文BC → OrderConfirmedPublic{OrderID, SeatNo, Items}
                    ↓ EventBus
厨房BC の subscriber → event.Items を直接使って調理開始
```

- 採用箇所: StartCookingUsecase
- メリット: シンプル。問い合わせ不要
- 適用条件: 必要データが少ない（5-10フィールド以下）

### 方法C: ローカルコピー（状態転送）

イベントで受け取ったデータをローカル DB に蓄積し、必要な時に読む。

```
注文BC → OrderCreated / OrderConfirmed 等のイベント
                    ↓ EventBus
厨房BC の subscriber → order_board テーブルに Upsert

別のタイミング:
  厨房スタッフがボード画面を開く
  → order_board テーブルから SELECT
  → 注文BC への問い合わせ不要
```

- 採用箇所: OrderBoard（横断 Projection）
- メリット: 複数イベントのデータを統合できる。発信元が落ちても動ける
- 適用条件: 同じデータを何度も参照する、横断ビューが必要

### 使い分けの判断基準

| 基準 | 方法B | 方法C |
|------|------|------|
| データ量 | 少ない（5-10フィールド） | 多い or 複数イベント統合 |
| 参照回数 | 1回（イベント受信時のみ） | 複数回（画面表示等） |
| 発信元依存 | なし | なし |
| 実装コスト | 低 | 中（キャッシュテーブル管理） |

## マイクロサービス移行時の変更点

| 要素 | モノリス（現在） | マイクロサービス |
|------|-------------|-------------|
| EventBus | メモリ内（Go関数呼び出し） | Kafka / RabbitMQ |
| イベント形式 | Go struct | JSON |
| Outbox | InMemory | DB テーブル + Relay Service |
| contract/ | 同一プロセス内で共有 | 共有ライブラリ or Protobuf |
| Projection | 同じ DB 内 | サービスごとの独立 DB |
